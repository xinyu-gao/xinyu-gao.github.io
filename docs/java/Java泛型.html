# 泛型程序设计

泛型程序设计（Generic programming）意味着编写的代码可以被很多不同类型的对象所重用。

# 1. 继承实现

在 Java 中增加范型类之前， 泛型程序设计是用继承实现的。

好处：

  • 使得程序具有更好的可读性和安全性。

缺点：

  • 当获取一个值时必须进行强制类型转换。
  • 没有错误检査，可以向数组列表中添加任何类的对象。
public class ArrayList{ //  增加范型类之前的ArrayList类，
	private Object[] elementData; 
	public Object get(int i) { . . . }
	public void add(Object o) { . . . } 
}

ArrayList files = new ArrayList(); 
String filename = (String) files.get(O); // 强制类型转换
files.add(new File("...")); // 添加其他类的对象

# 简单泛型类

# 1. 概念

一个泛型类（ generic class ）就是具有一个或多个类型变量的类。

TIP

简单泛型类：只关注泛型，不关心数据存储的细节。

在 Java 库中， 类型变量使用大写形式，并分别具有以下意义：

  • E 表示集合的元素类型；
  • K 和 V 分别表示表的关键字与值的类型；
  • T（ 需要时还可以用临近的字母 U 和 S ）表示“ 任意类型”。

# 2. 使用示例

MyTest 类引人了一个类型变量 T，用尖括号（ < > ）括起来，并放在类名的后面。

public class MyTest<T> {
    private T first;
    
    private T second;
    
    public MyTest(T first, T second) {
        this.first = first;
        this.second = second;
    }
    
    public T getFirst() { return first; }
    
    public void setFirst(T first) { this.first = first; }
    
    public T getSecond() {return second;}
    
    public void setSecond(T second) {this.second = second;}
}

# 泛型方法

TIP

泛型方法可以定义在普通类中，也可以定义在泛型类中。

类型变量放在修饰符（这里是 public static） 的 后面，返回类型（这里是void）的前面。

public class MyTest<T> {
    public static <T> void printData(T name){
        System.out.println("HelloWorld, "+name);
    }
}

当调用一个泛型方法时, 在方法名前的尖括号中放入具体的类型：

MyTest.<String> printData("yu");
1

在这种情况（实际也是大多数情况）下，方法调用中可以省略类型参数。

编译器有足够的信息能够推断出所调用的方法。

# 类型变量的限定

类或方法可能需要对类型变量加以约束。

对类型变量 T 设置限定：

public static  <T extends Comparable> T getFirst(T[] a){
    return a[0];
}
1
2
3

一个类型变量或通配符可以有多个限定， 例如：

T extends Comparable & Serializable

TIP

限定类型用 “ &” 分隔，而逗号用来分隔类型变量。

public static <T extends Comparable & Serializable, U extends Comparable> T min(T[] a ,U[]c){
    // . . .
    return a[0];
}
1
2
3
4

注意

可以根据需要拥有多个接口超类型，但限定中至多有一个类，且它必须是限定列表中的第一个。

使用关键字 extends 的原因：

表示 T 应该是绑定类型的子类型 （subtype)。 T 和绑定类型可以是类， 也可以是接口。选择关键字 extends 的原因是更接近子类的概念， 并且 Java 的设计者也不打算在语言中再添加 一个新的关键字（如 sub)。

# 类型擦除

# 1. 类型擦除简介

虚拟机不知道泛型的存在，所有的工作都是编译器做的。即虚拟机中没有泛型，只有普通的类和方法。

类型擦除是编译器做的。

无论何时定义一个泛型类型， 都自动提供了一个相应的原始类型 （ raw type ）。

# 2. 擦除规则

原始类型的名字就是删去类型参数后的泛型类型名。

擦除（ erased ) 类型变量，并替换为限定类型。

若有限定，原始类型用第一个限定的类型变量来替换

若无限定，用 Object 替换。

# 3. 代码示例

编译器执行的代码：

public class MyTest<T> {
    
    private T first;
    
    private T second;

    public MyTest(T first, T second) {
        this.first = first;
        this.second = second;
    }
    public static  <T extends Comparable> T getFirst(T[] a){
    	return a[0];
	}
    public T getFirst() { return first; }
    
    public void setFirst(T first) { this.first = first; }
    
    public T getSecond() {return second;}
    
    public void setSecond(T second) {this.second = second;}

}

# 翻译泛型

翻译泛型是编译器做的。

# 1. 翻译泛型表达式

MyTest<String> myTest =new MyTest<>("hello", "world");
String str = myTest.getSecond();
1
2

编译器把这个方法调用翻译为两条虚拟机指令：

  • 对原始方法 MyTest.getSecond() 的调用。

  • 将返回的 Object 类型强制转换为 String 类型。

# 2. 翻译泛型方法

有时候，类型擦除与多态可能发生冲突。

例如某个方法：

public class SonTest extends MyTest<String>{
	public void setSecond(String second) { . . . }
}
1
2
3

但其中还存在着从 MyTest 类继承的方法（即上述的方法并没有覆盖掉父类的方法）：

public void setSecond(Object second)
1

两个方法发生了冲突。

要解决这个问题， 就需要编译器在 SonTest 类中生成一个 桥方法（bridge method):

public void setSecond(Object second) { 
    setSecond((String) second); 
}
1
2
3

但是又出现了个问题，出现了方法签名相同的两个方法。

注意

我们绝对不能编写出方法签名一样的多个方法 。

一旦编译器通某种方式自己编译出方法签名一样的两个方法 （只能编译器自己来创造这种奇迹，我们程序员却不能人为的编写这种代码）。 JVM 还是能够分清楚这些方法的，前提是需要返回类型不一样。

# 约束与局限性

使用 Java 泛型时需要考虑一些限制，大多数限制都是由类型擦除引起的：

  1. 不能用基本类型实例化类型参数。

    没有 Pair<int>, 只 有 Pair<Integer>。

  2. 运行时类型查询只适用于原始类型。

    试图查询一个对象是否属于某个泛型类型时，倘若使用 instanceof 会得到一个编译器错误；getClass 方法总是返回原始类型。

  3. 不能实例化参数化类型的数组。

    不可以这样做： Pair[] table = new Pair[10];

# 泛型类型的继承规则

继承规则：

  • 无论 S 与 T 有什么联系 ，通常， Pair<S> 与 Pair<T> 都没有什么联系。
  • 永远可以将参数化类型转换为一个原始类型。

例如：

Student 是 Person 的一个子类，但是MyTest<Student> 不是 MyTest<Person> 的一个子类。

MyTest<Student> 是原始类型 MyTest 的一个子类型。在与遗留代码衔接时，这个转换非常必要。

# 通配符类型

# 1. 普通泛型缺点

public static void printData(MyTest<People> p)
1

根据泛型类型的继承规则，不能将 MyTest<Student> （Student 是 People 的子类 ）传递给这个方法，这一点很受限制。

固定的泛型类型系统有缺点，因此 Java 的设计者发明了一种巧妙的（仍然是安全的）“解决方案”：通配符类型。

# 2. 通配符的子限定类型

 public static void printData(MyTest<? extends People> p)
1

通配符类型中，允许类型参数变化：

例如，可以将 MyTest<Student> 传递给这个方法，因为 MyTest<Student> 是 MyTest<? extends People> 的子类。

作用：这个通配符限制 People 的所有子类型。

TIP

MyTest<? extends People> ，其方法似乎是这样的（这不是真正的 Java 语法，仅仅为了描述编译器知道什么）：

void setFirst(? extends People); // 不可
? extends People getFirst(); // 可以
1
2

这样将不可能调用 setFirst 方法。编译器只知道需要某个 People 的子类型，但不知道具体是什么类型。它拒绝传递任何特定的类型。

而 getFirst 就不存在这个问题： 将 getFirst 的返回值赋给一个People 的引用完全合法。

注意

带有子类型限定的通配符可以从泛型对象读取，但不能写入。

# 3. 通配符的超类型限定

通配符限定与类型变量限定十分类似，可以指定一个超类型限定（ supertype bound ）? super Student。

作用：这个通配符限制为 Student 的所有超类型。

TIP

MyTest<? super Student> ，其方法似乎是这样的（类比上面）：

void setFirst(? super Student); // 可以
? super Student getFirst(); // 不可
1
2

编译器无法知道 setFirst 方法的具体类型， 因此调用这个方法时不能接受类型为 People 或 Object 的参数。 只能传递 Student 类型的对象，或者某个子类型对象。

如果调用 getFirst，不能保证返回对象的类型，只能把它赋给一个 Object。

注意

带有超类型限定的通配符可以向泛型对象写入，但不能读取。

换个说法：

  • 如果希望只读取，不读取，就使用 ? extends
  • 如果希望只写入，不取出，就使用 ? super
  • 如果希望既能写入，又能读取，就不要用通配符。

# 4. 无限定通配符

public static boolean hasNulls(MyTest<?> p)
1

其等价于:

public static  <T> boolean hasNulls(MyTest<T>  p)
1

TIP

带有通配符的版本可读性更强。
